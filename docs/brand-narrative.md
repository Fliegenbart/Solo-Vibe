# Brand Narrative

This document defines the canonical narrative for Solo-Vibe. All copy across the repo, landing page, and marketing must align with these blocks.

## Canonical headline

**Your AI code deserves a home.**

## Canonical subheadline

Solo-Vibe safely stores your AI-generated code and deploys it to your own server. No Git. No lock-in. No developer skills required.

## Canonical product explanation (3 steps)

1. **Upload** — Bring your AI-generated code into Solo-Vibe
2. **Save & version** — Every snapshot is preserved. Roll back anytime.
3. **Go live** — One click to your own server. Health-checked. Yours.

## What it is / what it is not

**Solo-Vibe is:**
- A safe home for your AI-generated code
- A simple bridge from code to the internet
- An open tool — your code, your server, your export, always

**Solo-Vibe is not:**
- A code editor (use Cursor, VS Code)
- An AI coding tool (use ChatGPT, Claude, Lovable)
- A hosting provider (you bring your own server)
- A GitHub replacement (no teams, no branches, no pull requests)

## Core promise

**Ownership.** Your code belongs to you. Export anytime. No proprietary formats.
**Reversibility.** Every version is saved. Roll back to any point. Nothing is permanent.
**Simplicity.** No Git commands. No terminal. No infrastructure knowledge required.

## Wedge statement

*Build in Lovable. Ship with Solo-Vibe.*

Solo-Vibe is the missing bridge between AI-generated code and the internet.

## Approved copy blocks

### For GitHub (README, repo description)

> The open workbench that safely stores your AI-generated code and deploys it to your own server — no Git, no lock-in, no developer skills required.

### For landing page hero

> Your AI code deserves a home.

### For social/sharing

> Built something with AI? Solo-Vibe gets it on the internet. On your server. One click.

### For technical audience

> Solo-Vibe uses Git internally for versioning, SSH for deployment, and AES-256-GCM for secret encryption. The user sees none of this. They see: upload, save, go live, roll back.

## Tone guidelines

- Calm confidence, not hype
- Concrete, not abstract
- Honest about limitations
- No "revolutionary," "game-changing," "disruptive"
- No developer jargon in user-facing copy
- Technical accuracy in docs and architecture

See [language-system.md](language-system.md) for the full vocabulary and tone rules.
